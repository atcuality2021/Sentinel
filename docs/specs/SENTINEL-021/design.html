Approach

Name

Write-time deterministic reconciliation: resolve in the same transaction the conflict is detected.

Description

The defect is the gap between "conflict detected" and "conflict resolved": during it, both contradicting entries are quarantined=0, so recall() serves both. We close the gap by resolving at the moment of detection. In write()'s G-10 block, the instant a rival is found, a pure _pick_winner(a, b) chooses the live head; the loser is quarantined and gets a superseded_by link to the winner; the conflict row is written already-resolved (never 'open'). Because the live set is repaired before the transaction returns, recall() needs no change — the existing quarantined=0 filter does the read-side work. A reconcile_open_conflicts() curator applies the same pure policy to the historical backlog, wrapped in a golden-question rollback so a bad demotion can never degrade a real recall. The whole path is deterministic and model-free, so it is compliance-safe regardless of the repo's compliance mode and trivially unit-testable.

Alternatives Considered

#AlternativeWhy Rejected
1Recall-time collapse — leave writes as-is, dedup contradictions in recall() Puts policy on the hot read path (every recall re-runs it), and leaves the DB in a permanently-contradicted state that every other reader (UI, exports, curator) must also defend against. Deferred to a follow-up safety net only (spec OQ-2 / Out of Scope).
2LLM-judged "which fact is true" Non-deterministic, costs tokens, and violates on_prem_required/cost posture for what is fundamentally a recency+trust ordering. The deterministic policy is sufficient and auditable.
3Hard-delete the loser Destroys the audit trail and the wake-up path. Sovereignty/provenance is the moat — we keep the loser quarantined and linked, never gone (architecture §6: "never delete silently").
4Keep resolve_conflict(keep="a") and just schedule it The arbitrary keep side is the bug, not the trigger. Auto-calling an arbitrary policy automates the wrong outcome. We must fix the policy first.

API Contract Changes

One new pure function and one new method. No signature changes to existing callers (additive).

# pure, deterministic, no DB / no LLM / no clock — order-independent
def _pick_winner(a: MemoryEntry, b: MemoryEntry) -> tuple[MemoryEntry, MemoryEntry]:
    """Return (winner, loser). Tie-break order (AC2):
      1. newer created_at wins
      2. boundary trust: PRIVATE > PUBLIC          (OQ-1, sovereign posture)
      3. higher SM-2 strength, then access_count
      4. stable lexicographic id (final deterministic tiebreaker)
    """

# curator entry point; fail-soft; wrapped in golden-question rollback
def reconcile_open_conflicts(self, *, golden_top_n: int = 20) -> dict:
    """Resolve every status='open' memory_conflicts row via _pick_winner.
    Snapshot recall() answers for the top-N most-accessed entities, reconcile,
    re-recall; roll back (un-quarantine) any entity whose answer degraded.
    Returns {resolved, rolled_back, flagged}.
    """

Note: resolve_conflict(keep="a") stays for manual/UI use but is no longer the only path; the write-time resolver and the curator both route through _pick_winner so there is one policy, not two.

Security & Compliance

  • Compliance: the resolver is deterministic and model-free — no external/LLM call, so it is safe under any compliance mode (no ADR needed).
  • Boundary integrity: PRIVATE > PUBLIC tie-break (OQ-1) means web-sourced PUBLIC data can never silently overwrite an operator-vetted PRIVATE fact of equal recency. One-line ADR note recorded.
  • No data loss: the loser is quarantined, not deleted, and linked via superseded_by. Fully reversible (un-quarantine + clear link).
  • Fail-soft preserved: the G-10 block is wrapped in try/except: pass today (store.py:562) — auto-resolve stays inside that guard so a resolver bug can never break a write.

Performance Considerations

Write path: the rival lookup already runs (line 548). We add one _row_to_entry on the rival + a pure comparison + the same UPDATE/INSERT the manual resolver already does — O(1), same transaction, negligible. Recall path: unchanged (no new work). Curator pass is O(open conflicts) and runs off the hot path on the cadence only.

Rollout Plan

  1. Land _pick_winner + unit tests (no behavior change yet — pure fn).
  2. Add the migration + schema field + row/insert threading (additive, dormant).
  3. Wire auto-resolve into write()'s G-10 block (new contradictions self-heal).
  4. Add reconcile_open_conflicts() + golden-question guard; clear the backlog.
  5. Surface the count in the UI; wire the curator cadence.

Rollback

Revert the write() edit → new contradictions log 'open' again (pre-fix behavior). The superseded_by column is harmless if unused. To restore demoted entries: UPDATE memory_entries SET quarantined=0, superseded_by=NULL WHERE superseded_by IS NOT NULL. No data was destroyed, so rollback is lossless.
