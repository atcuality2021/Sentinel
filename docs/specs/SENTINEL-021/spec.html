Problem

When Sentinel learns a fact that contradicts an existing one about the same entity, the G-10 conflict detector (store.py:542) logs a memory_conflicts row but quarantines neither entry. Resolution is manual and arbitrary — resolve_conflict(keep="a") (store.py:609) defaults to keeping an unspecified side, and nothing calls it automatically. Because recall() (store.py:420) returns every non-quarantined entry, in the window between detection and a human resolving it, recall serves both contradicting findings. A research output can therefore cite two opposite "facts" about one entity, directly undermining the provenance/sovereignty guarantee that is Sentinel's moat.

User Impact

After this ships, a research run grounds on one reconciled finding per entity+topic — the newer, more-trusted one — instead of an accreting pile of contradictions. The losing finding is preserved (quarantined, linked) for audit, never silently destroyed. The memory/accounts UI shows "reconciled — N superseded" so the operator can see what was demoted rather than discovering a contradiction in a deliverable. Internal/operational: conflict backlog stops growing unbounded.

Acceptance Criteria

  • OPEN AC1 — Given two non-quarantined entries for the same entity+boundary sharing the 40-char topic prefix but differing in content, after the second write recall() returns exactly one — the policy winner.
  • OPEN AC2 — _pick_winner(a, b) is a pure, deterministic function: newer created_at wins; tie → boundary trust (PRIVATE > PUBLIC, OQ-1); tie → higher SM-2 strength/access_count; final tie → stable (lexicographic id). Unit-tested in both argument orders (same winner regardless of order).
  • OPEN AC3 — The losing entry is quarantined, not deleted, and a new superseded_by field points at the winner's id (audit + wake-up path).
  • OPEN AC4 — Exact-duplicate writes still reinforce via SM-2 (store.py:517 path) and do not create a conflict — no regression.
  • OPEN AC5 — reconcile_open_conflicts() auto-resolves every status='open' backlog row via the policy, wrapped in a golden-question check (snapshot N entity recalls → reconcile → re-recall) that rolls back any entity whose recall answer degraded.
  • OPEN AC6 — The superseded_by column is added via an idempotent migration (_MEMORY_MIGRATIONS, store.py:293); a pre-migration DB loads and runs unchanged (defensive _col read).
  • OPEN AC7 — The memory/accounts UI surfaces a reconciliation count ("N superseded"); no conflict is dropped silently.

Out of Scope

  • Target A — the agent's .claude write-hook + curator cadence (separate track, no SENTINEL id; shares the _pick_winner policy + golden-question check conceptually).
  • Cross-entity reconciliation / entity-merge. recall() is entity-keyed; topic-prefix conflicts are scoped within one entity+boundary.
  • LLM-based semantic conflict detection. v1 policy is deterministic and pure (no model call, compliance-safe).
  • Recall-time collapse safety net — deferred per OQ-2. v1 is write-time resolution only; un-logged pre-fix near-dups are handled by the curator backlog pass (AC5), not by a recall-path collapse. Recall-time collapse is a follow-up if races/leftovers warrant it.
  • The v-next reusable-workflow program (SENTINEL-017…020).

Open Questions — RESOLVED 2026-06-14

  • OQ-1 → PRIVATE > PUBLIC. On a created_at tie, the PRIVATE (operator-vetted) entry wins over PUBLIC (web-sourced). Baked into AC2. Aligns with the sovereign/regulated posture. Worth a one-line ADR note in design.
  • OQ-2 → write-time resolution only. Resolve in the same transaction the conflict is logged; no recall-time collapse in v1 (moved to Out of Scope). Backlog handled by the curator (AC5).
  • OQ-3 → auto-derived. The golden-question set is auto-derived from the top-N most-accessed entities at curator run time (no hand-maintained list).
