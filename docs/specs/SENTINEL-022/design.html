Ground Truth (code as it is today)

The relevant machinery already exists; SENTINEL-022 re-wires three small parts of it. Anchors, read live 2026-06-14:

  • src/sentinel/agent/modes/spec.py:377-409 — the step.tool == "search" branch of build_agents. It (a) resolves an effective provider, (b) builds the search tool, (c) appends the KB tool, (d) tools.extend(build_mcp_toolsets(cfg, spec.domain, cloud_allowed=...)). Crucially, lines 383-390 already contain a gemini → duckduckgo swap — but its guard is _resolved == "vllm", so it only fires on the sovereign/vLLM path. On the cloud path (cloud_allowed=True, backend resolves to gemini), _eff_provider stays "gemini" and get_search_tool("gemini") returns the built-in google_search — which then collides with the MCP function-toolsets attached 12 lines later.
  • src/sentinel/tools/public/web_search.py:348-351 — get_search_tool: provider == "gemini" returns the ADK built-in google_search; every other provider returns a _make_function_tool closure. Built-in grounding tools cannot be combined with function-calling in one Gemini agent → 400 INVALID_ARGUMENT.
  • src/sentinel/tools/mcp_registry.py:29-48 — build_mcp_toolsets: returns [] when cloud_allowed=False (the sovereign gate); otherwise one McpToolset per enabled + secret-configured server whose domains match (empty = all). searchapi + firecrawl both have domains=[], so both attach to product_research.
  • src/sentinel/config/defaults.py:697-717 — _P_PRODUCT_RESEARCH_PUBLIC: the “ATTACK SEQUENCE” prompt. It directs only generic site:flipkart.com / site:amazon.in web searches; it never names google_shopping_search or google_product.
  • src/sentinel/config/defaults.py:1010-1018 — the searchapi MCPServerConfig: transport="http", url_env="SEARCHAPI_MCP_URL", tool_filter=[] (dashboard-curated), domains=[], and a vague description (“SearchAPI.io engines — toolset curated in the dashboard”) that gives the model no reason to prefer the shopping tool.
  • src/sentinel/config/schema.py:268-270 — SearchConfig.provider (default gemini) and SearchConfig.onprem_fallback (default duckduckgo): an already-validated field whose documented meaning is “the non-cloud provider to use when policy forbids Gemini but the configured provider is still gemini.” This design reuses it as the swap target.
  • src/sentinel/memory/store.py — a family of single-responsibility SQLite stores (MemoryStore, RunStore, TelemetryStore, …) sharing one DB file. All tables are created centrally in one _SCHEMA string (lines 39-265) applied by _ensure_schema() via executescript (line 311); every store's __init__ just calls _ensure_schema(self.path). Column additions to existing tables go through a separate _apply_column_migrations / ALTER TABLE ADD COLUMN path (line 300-313, e.g. SENTINEL-021's superseded_by). AC4 adds a new table, so it belongs in _SCHEMA, not the column-migration path.

Two levers, both isolated by live experiment (recorded in spec Evidence): the built-in/MCP collision alone moves the query 0→0; removing it (provider→duckduckgo) moves it 0→3 sources but products stay 0 because the agent never calls the shopping tool. The fix needs both AC1 (drop the built-in when MCP attaches) and AC2 (name the shopping tool).

Approach

Name

Widen the existing provider-swap guard, a shopping-first prompt, and a deterministic code-owned price-discovery cascade tool (dev decision 2026-06-14: build up front for guaranteed determinism, not measurement-gated).

Description

The model already proved it skips google_shopping_search even when the tool is on its menu (spec Evidence: 0→3 sources but 0 products). So the price path is taken out of the model's hands: a single find_deals function-tool owns a deterministic shopping→SERP→Firecrawl cascade, and the prompt directs the agent to use it for all pricing. The collision fix (AC1) and shopping-first prompt (AC2) still land — they make the agent's review/comparison searches work and remove the 400 — but the priced-deal guarantee now lives in tested code, not in prompt compliance.

Layer 1 — collision fix (AC1, AC6). In the step.tool == "search" branch, build the MCP toolsets first, then choose the provider with knowledge of whether function-toolsets are present:

mcp_tools = []
try:
    from sentinel.tools.mcp_registry import build_mcp_toolsets
    mcp_tools = build_mcp_toolsets(cfg, spec.domain, cloud_allowed=cloud_allowed)
except Exception:
    mcp_tools = []          # fail-soft, unchanged

_eff_provider, _keep_mcp = _resolve_search_tools(
    requested=search_provider, resolved=_resolved, pin_gemini=_pin,
    has_mcp_funcs=bool(mcp_tools), onprem_fallback=cfg.search.onprem_fallback,
)
if not _keep_mcp: mcp_tools = []
tools = [get_search_tool(_eff_provider, results=cfg.search.results, ...)]
if kb_tool is not None: tools.append(kb_tool)
tools.extend(mcp_tools)

_resolve_search_tools(...) is a new pure helper (no I/O) returning (effective_provider, keep_mcp), unit-tested across every branch. Build-time discovery (important): with SearchAPI + Firecrawl configured (domains=[] ⇒ they attach to every domain), the built-in/function-tool collision is not unique to product_research — every cloud + gemini research step was already attaching the built-in google_search alongside 2 MCP function-toolsets (the same 400). To keep SENTINEL-022 scoped, the swap fires only for non-pinned steps: product_research (not pinned) → onprem_fallback, keep MCP; the three pin_gemini modes (competitor / client / self_profile) are left exactly as-is (built-in + MCP), so no existing test or behavior changes. Resolving the pinned modes' latent collision is a tracked follow-up (TD-SENTINEL-022), not bundled here. Branch matrix: non-gemini → as-is; gemini+vLLM → fallback; gemini+pinned → unchanged; gemini+unpinned+MCP → fallback; gemini+unpinned+no-MCP → built-in.

Layer 2 — shopping-first prompt (AC2). Rewrite _P_PRODUCT_RESEARCH_PUBLIC so steps 1-2 of the attack sequence are google_shopping_search (criteria → priced listings, capture each product_token) then google_product (token → multi-seller offers + price history), and the generic site: web searches become the fallback steps 3+. Enrich the searchapi server description so the tool-caller can see it exposes shopping engines. Prompt-content unit test asserts both tool names appear.

Layer 3 — deterministic cascade tool (AC3) + learning signal (AC4). The core of the dev's chosen approach. Three pieces:

  • Programmatic MCP shopping client — new src/sentinel/tools/shopping_client.py. Calls SearchAPI's google_shopping_search and google_product over the MCP transport (SEARCHAPI_MCP_URL) using the mcp SDK's streamablehttp_client + ClientSession — not a hand-rolled HTTP client, and distinct from ADK's McpToolset (which only exposes tools to the agent). The token authenticates only over MCP, so this is the one correct programmatic path (the REST api_key 401s). Async; returns normalized priced rows {title, price, source_url, seller, product_token}.
  • Cascade orchestrator + pure policy — new src/sentinel/tools/shopping.py. build_deal_search_tool(...) returns an ADK function-tool (find_deals(query)) whose body deterministically runs: shopping → if shopping_results_thin(rows, min_priced=3) → SERP fallback (get_search_tool(onprem_fallback)) → if still thin → Firecrawl deep-read of the top candidate. The three legs are injected callables, so the cascade logic is unit-tested with fakes — both paths (shopping-hits → SERP/Firecrawl not called; shopping-thin → fallback fires) — with zero network. shopping_results_thin and classify_query_class are the pure helpers (OQ-3: “thin” = <3 rows with a non-empty price).
  • ToolPreferenceStore (AC4) — a dedicated tool_preference table (query_class PK → winning_tool, updated_at). The cascade records which leg produced priced rows per coarse query-class; on the next run the tool reads it to try the historically-winning leg first. Degradation-safe: any read error → no preference → default cascade order. Decoupled from MemoryStore so it never entangles with SENTINEL-021 reconciliation.

Wiring (AC1, AC6). In product_research's search step, the find_deals cascade tool is attached only when cloud_allowed=True (it calls SearchAPI — same sovereign gate as build_mcp_toolsets). The generic search tool (for review/comparison searches) and the Firecrawl McpToolset stay; the gemini built-in is dropped whenever any function-tool is present (AC1). On the sovereign path no cascade tool, no SearchAPI — the on-prem function-tool search is unchanged (AC6).

Layer 4 — live proof (AC5). Run the API-only e2e on the isolated server for a real buying query and assert products_found ≥ 5 with non-empty price+source_url, populated value_ranking, and a winner with a sourced rationale. The SERP-dependent assertion is skipped/marked when the MCP is unreachable (no flaky CI).

Alternatives Considered

AlternativeWhy not (now)
Default product_research to a function-tool provider in config (narrow, config-only AC1 fix) Fixes only one domain. Any other MCP-using domain (e.g. a future commerce or research mode) hits the identical built-in/MCP collision. The general guard-widen is the same code size and protects all domains. Resolves OQ-1 → general fix.
Prompt + policy only (name the shopping tool, no code cascade); measure with AC5 and add a cascade tool only if the model still skips shopping Rejected by dev 2026-06-14. Lighter, but it leaves the priced-deal guarantee dependent on model tool-compliance — and the spec Evidence already shows the model skipping google_shopping_search when it's merely available. The dev chose certainty over the lighter staged path.
Code-level cascade tool (CHOSEN): a single find_deals(query) function-tool owning a deterministic shopping → SERP → Firecrawl cascade Dev decision 2026-06-14. Strongest guarantee — removes model tool-choice from the price path entirely. The duplication concern (AP#1/#2) is avoided by reusing the mcp SDK's streamable-http client for the programmatic call rather than hand-rolling JSON-RPC; the cascade orchestration is genuinely new behavior (ordered fallback + learning), not a re-implementation of McpToolset's agent-exposure role.
Store the learning signal in MemoryStore (reuse product memory) Entangles tool-preference rows with SENTINEL-021's entity/topic reconciliation (superseded_by, quarantine, live-head). A bad tool-preference write could trip reconciliation; recall could surface a preference as a “finding.” A dedicated table keeps the two concerns isolated. Resolves OQ-2 → dedicated lightweight store.
Playwright/Chromium on the server as primary discovery Rejected in spec Out-of-Scope: datacenter-IP bot-walls, per-page latency × tool-call budget, brittle selectors, still egresses. Firecrawl covers single-page deep-read; Playwright reserved for true interaction in a future task.
Enable Google Custom Search JSON API / stand up SearXNG Alternative providers, unnecessary once the shopping tool is wired. CSE is currently 403 (API disabled on the GCP project); not on the critical path.

Data Model Changes

One new table (AC4), additive, in the existing memory SQLite DB:

CREATE TABLE IF NOT EXISTS tool_preference (
    query_class  TEXT PRIMARY KEY,   -- coarse bucket, e.g. "product_research:shopping"
    winning_tool TEXT NOT NULL,      -- e.g. "google_shopping_search"
    updated_at   TEXT NOT NULL       -- ISO-8601, UTC
);

query_class is a coarse, bounded key (domain + intent), not the raw query — this caps row growth and makes the “learn” signal generalize across similar buying queries. Creation path: the CREATE TABLE IF NOT EXISTS block is appended to the centralized _SCHEMA string (after the personas block, ~store.py:259) so _ensure_schema's executescript creates it on first DB open — the same path as every existing table. It is not added to _MEMORY_MIGRATIONS (that mechanism is ALTER TABLE ADD COLUMN for existing tables). No backfill (absence ⇒ no preference ⇒ default order); no change to any existing table, so all current fixtures pass unchanged.

API Contract Changes

None external. No new HTTP route, request, or response shape. The ProductResearch / ProductOption output schema is unchanged — the whole point is that the existing price / source_url / winner / value_ranking fields finally get populated. Internal-only new symbols: _resolve_search_tools(), shopping_results_thin(), classify_query_class(), ToolPreferenceStore.

Security & Compliance

  • Sovereign posture (AC6) preserved by construction. build_mcp_toolsets returns [] when cloud_allowed=False, so has_mcp_funcs is False on the sovereign path → the provider swap never triggers → no built-in/MCP mix and no new cloud dependency on the on-prem path. A unit test pins this.
  • No new secrets, no secret values in config. Reuses SEARCHAPI_MCP_URL (already in .env, named-not-valued in config) and the existing searchapi server entry. No .env value is read into, logged by, or committed from any new code.
  • Compliance mode. No new external AI/cloud-LLM dependency is added — SearchAPI + Firecrawl already attach to product_research today; this task only changes which of the already-present tools the agent prefers. (Confirm the repo mode in AGENT_RULES.md § Compliance at Review; if an ADR is required for the existing cloud-search posture it already exists from SENTINEL-013/015.)
  • No new injection surface. Prompt rewrite keeps the existing _INJECTION_STANCE suffix; shopping results are treated as untrusted public findings exactly like SERP rows today.

Performance Considerations

  • Shopping-first should reduce tool turns vs the current generic 4-5 search attack sequence (one structured call returns ~40 priced rows vs several thin SERP calls). The existing cfg.search.max_calls budget (default 8) still caps the loop.
  • ToolPreferenceStore reads/writes are single-row, indexed by PK, off the hot path (one read at step-build, one write post-research). Negligible.
  • The deferred Layer-4 cascade tool, if ever built, would add at most one fallback SERP + one Firecrawl call only when shopping is thin — bounded by the same max_calls budget.

Rollout Plan

  • Pure code + config + one additive table; no migration risk to existing data, no API change. Ships on the feature/SENTINEL-022-shopping-search branch through the normal Attack Loop (Build → Review → Test → Ship).
  • Build order is bottom-up: the pure policy + MCP client + cascade orchestrator land and are unit-tested before the spec.py wiring, then AC2 prompt, then the AC5 live proof. The cascade tool is default scope (dev decision), not conditional.
  • Feature is effectively always-on for product_research on the cloud path; the sovereign path is unaffected (AC6).

Rollback

Revert the branch. The tool_preference table is additive and orphan-safe (nothing else references it); it can be left in place or dropped. No data migration to unwind.
