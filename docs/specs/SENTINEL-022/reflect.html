What Went Well

  • Bottom-up build order (Step 1 leaf → Step 7 live) kept each step independently testable. The cascade passed unit tests before it was ever wired into product_research.
  • The collision guard (AC1) was the highest-risk change — touching the shared agent_spec.py branch. Isolated to a cloud_allowed guard that is already present, so no new always-on surface was introduced.
  • ToolPreferenceStore uses SQLite (same DB, different table) — zero new infra dependency. Fail-soft on corrupt DB means a bad preferences row never crashes a research run.

What to Watch

  • Pinned-mode collision (TD-SENTINEL-022): AC1 guard applies to cloud/auto-select modes only. If a future mode pins a provider and attaches function toolsets, the google_search collision could resurface. Scoped to a tech-debt note; no user-facing risk today.
  • SearchAPI quota: google_shopping_search consumes SearchAPI credits per call. High-frequency product research (e.g., batch deal-finder) should rate-limit. Not in scope for this task.
  • Thin-result heuristic: shopping_results_thin() uses a threshold of <3 priced results. If SearchAPI degrades (partial response), the cascade falls back to SERP and the ToolPreferenceStore records SERP as the winner — degrading future runs. A minimum-confidence score on the preference record would prevent this regression.
