Problem

A real product_research run ("best laptop under 80000 INR with 16GB RAM and RTX graphics for video editing") returns zero products — products_found=0, sources=0, winner="No qualifying product found", 78 empty gaps. The mode is correctly designed (the product_research.public_research prompt carries a Flipkart/Amazon site: "attack sequence"; ProductOption.price/ source_url and ProductResearch.winner/value_ranking already model the "best deal"), but the search layer underneath it never returns priced e-commerce listings. Two root causes, both confirmed live:

  • Built-in/function-tool collision. The active search provider is gemini, so get_search_tool("gemini") returns the ADK built-in google_search (tools/public/web_search.py). Built-in grounding tools cannot be combined with function-calling in one Gemini agent → 400 INVALID_ARGUMENT. The product_research research step already attaches the firecrawl + searchapi MCP function-toolsets (build_mcp_toolsets), so the collision compromises the whole step (fail-soft → empty findings → 0 products).
  • The agent is never told to use the shopping tool. Even with the collision removed, the prompt's "attack sequence" is written for generic web search, and the SearchAPI server description is vague. The tool-calling model falls back to weak web search and never invokes google_shopping_search, which is the only path that returns structured prices.

Evidence (proven live, isolated server)

The data is one MCP call away. google_shopping_search (via the already-configured SearchAPI MCP, SEARCHAPI_MCP_URL) returned 40 live priced listings in 1.6s for "laptop 16GB RAM RTX 4050", gl=in — e.g. Lenovo LOQ ₹63,990 (Flipkart), Acer Nitro ₹75,999 (Reliance Digital), HP Victus ₹88,000 (amazon.in) — each with seller + product_token (which feeds google_product → multi-seller offers + pricing history).

The two-lever diagnosis is experimentally isolated. Switching the provider gemini → duckduckgo (function-tool, no built-in collision) moved the same query from 0 → 3 sources and lengthened runtime (35s → 49s = search actually ran) — but products_found stayed 0 because the agent used weak DuckDuckGo web search and never called google_shopping_search. Both levers are required.

Acceptance Criteria

  • OPEN AC1 — When MCP function-toolsets are attached to a research step, the agent is not given the built-in gemini google_search tool in the same agent (no 400 built-in/function-calling collision). Covered by a unit test asserting the tool list for a product-research step contains no built-in search alongside function-tools.
  • OPEN AC2 — The product_research.public_research prompt explicitly directs the agent to call google_shopping_search (criteria → priced listings) and then google_product (product_token → multi-seller offers) before any generic web search. Prompt-content test asserts both tool names are named.
  • OPEN AC3 — A search tool cascade exists: the step prefers the structured shopping tool, and falls back to web SERP / Firecrawl only when shopping results are thin/empty. The cascade is pure/deterministic and unit-tested in both the "shopping-returns-results" and "shopping-empty → fallback" paths.
  • OPEN AC4 — "Agent learns": the step records which tool yielded usable priced results per query class and prefers it on the next run (lightweight persisted signal). Degradation-safe — a bad signal never blocks a run. Unit-tested.
  • OPEN AC5 — With the SearchAPI MCP reachable, a product_research run for a real buying query yields products_found ≥ 5 with non-empty price + source_url, a populated value_ranking, and a winner with a sourced winner_rationale. Verified by a live e2e (API-only) on the isolated server; the SERP-dependent assertion is skipped/marked when the MCP is unreachable (no flaky CI).
  • OPEN AC6 — Sovereign posture preserved: when cloud_allowed=False, no external SearchAPI/Firecrawl toolset is offered (existing build_mcp_toolsets gate). No regression to the on-prem path; no new always-on cloud dependency added without an ADR.

Out of Scope

  • Flipkart Affiliate + Amazon PA-API direct MCPs — Phase 2 (dev decision 2026-06-14). Official per-retailer product/price feeds + affiliate-link monetization land as their own MCP servers later; this task ships working price discovery via the existing google_shopping_search engine first.
  • Playwright/Chromium-on-server as a primary search path. Rejected as primary (datacenter-IP bot-walls on Amazon/Flipkart, per-page latency × tool-call budget, brittle selectors, still egresses). Single-page deep-read stays with Firecrawl (already live); Playwright reserved for true interaction (login/click-through) in a future task if needed.
  • Enabling Google Custom Search JSON API / standing up SearXNG — alternative providers, not required once the shopping tool is wired. (CSE is currently 403: API disabled on the GCP project.)
  • Extending the shopping/deal flow to non-product domains.

Open Questions

  • OQ-1 — AC1 mechanism. Suppress the built-in google_search whenever MCP function-tools are present (general fix, all domains), or default product_research to a function-tool provider (narrow, config-only)? Lean to the general fix so other MCP-using domains don't hit the same collision — confirm in design.
  • OQ-2 — "learn" store. Reuse the existing product-memory store (a tool-preference entry per query class) or a small dedicated table? Lean to the lightest persisted signal that does not entangle with SENTINEL-021 reconciliation — confirm in design.
  • OQ-3 — fallback trigger. What counts as "thin" shopping results (e.g. < N results, or zero priced rows) before falling back to web SERP/Firecrawl?
