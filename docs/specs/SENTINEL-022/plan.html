Build Sequence

Each step = one logical commit. TDD: RED → GREEN in the same step. Run pytest before proceeding. Bottom-up: leaf modules (store, policy, client, orchestrator) land + unit-test before the shared spec.py wiring, so the highest-risk edit goes in last against green foundations. Step 1 — ToolPreferenceStore + tool_preference table AC4

Rationale: The learning signal is a leaf with no deps — build it first so the cascade (Step 4) can write to a real store. Dedicated table keeps it decoupled from MemoryStore / SENTINEL-021 reconciliation.

Files: src/sentinel/memory/store.py — (a) append the CREATE TABLE IF NOT EXISTS tool_preference (...) block to the centralized _SCHEMA string (after the personas block, ~line 259), so the shared _ensure_schema executescript creates it — not a per-store CREATE and not the _MEMORY_MIGRATIONS/ALTER path (that is for adding columns to existing tables); (b) new ToolPreferenceStore whose __init__ calls _ensure_schema(self.path) like every other store, with get(query_class) -> str | None and record(query_class, winning_tool) (upsert), both degradation-safe (missing table / bad row → None, never raises).

Tests — tests/test_tool_preference_store.py: fresh-DB round-trip through _ensure_schema — construct a real ToolPreferenceStore against a temp db_path(), assert the tool_preference table exists and record→get round-trips (this is the only test that catches a wrong creation path); upsert overwrites; unknown class → None; corrupted/absent table → None (no exception). Gate: pytest tests/test_tool_preference_store.py — GREEN.
Step 2 — Pure policy shopping_results_thin() + classify_query_class() AC3

Rationale: The deterministic pivot the cascade turns on, with zero I/O — pinned independently of any network so the cascade's branch logic is provably correct.

Files: new src/sentinel/tools/shopping.py — shopping_results_thin(results, *, min_priced=3) -> bool (OQ-3: “thin” = <3 rows with a non-empty price); classify_query_class(domain, target) -> str (coarse, bounded key, e.g. "product_research:shopping").

Tests — tests/test_shopping_cascade.py (policy half): 3 priced rows → not thin; 2 priced + 5 priceless → thin; empty → thin; class key is stable + bounded for varied queries. Gate: pytest tests/test_shopping_cascade.py -k thin or class — GREEN.
Step 3 — Programmatic SearchAPI MCP shopping client AC3 · AC5

Rationale: The cascade's real shopping leg. Isolated here so its transport quirks (MCP-only auth) are contained and unit-tested with a mocked session.

Files: new src/sentinel/tools/shopping_client.py — async call_shopping_search(query, *, gl="in") and async call_google_product(product_token) via the mcp SDK streamablehttp_client(SEARCHAPI_MCP_URL) + ClientSession (initialize → call_tool). Normalize to {title, price, source_url, seller, product_token}. Fail-soft: transport/parse error → [] (logged, never raises into the agent loop). Reads the URL via os.getenv; the secret value is never logged.

Tests — driven by asyncio.run(...) with a fake ClientSession (monkeypatched): a canned tools/call payload normalizes to priced rows; a raising session → []; missing SEARCHAPI_MCP_URL → []. No live call here (that's AC5). Gate: pytest tests/test_shopping_client.py — GREEN.
Step 4 — Cascade orchestrator build_deal_search_tool() / find_deals AC3 · AC4

Rationale: Assembles Steps 1-3 into the deterministic find_deals function-tool the agent calls. Legs injected so the cascade is tested with fakes (no network), then bound to the real client/SERP/Firecrawl at construction.

Files: src/sentinel/tools/shopping.py — build_deal_search_tool(cfg, *, shopping_leg, serp_leg, firecrawl_leg, prefs) -> Callable returning an async find_deals(query). Logic: shopping_leg → if shopping_results_thin → serp_leg → if still thin → firecrawl_leg; record the winning leg in prefs (ToolPreferenceStore) keyed by classify_query_class; on next call, try the recorded winner first.

Tests — tests/test_shopping_cascade.py (orchestrator half), via asyncio.run with fake legs: (path A) shopping returns 5 priced → serp/firecrawl not called, store records google_shopping_search; (path B) shopping thin → serp called → returns rows, firecrawl not called; (path C) both thin → firecrawl called with a concrete top-candidate URL/query (assert the input it receives, not merely that it was called — avoids a vacuous spy pass, AP#10); (learn) second call with a recorded winner tries it first. Assert leg-call counts via spies. Gate: pytest tests/test_shopping_cascade.py — GREEN.
Step 5 — Wire into product_research: provider guard + cascade attach AC1 · AC3 · AC6

HIGHEST-RISK STEP. Touches the shared step.tool == "search" branch used by every domain. Land it last, against green foundations, with a sovereign regression test pinned.

Files: src/sentinel/agent/modes/spec.py — (a) new pure _search_provider_for_step(requested, resolved, has_mcp_funcs, onprem_fallback) widening the existing vLLM-only swap (lines 387-390) to “any step with MCP function-tools”; (b) reorder the branch to build mcp_tools first; (c) when spec.domain == "product_research" and cloud_allowed, append the find_deals cascade tool (bound to the real legs).

Tests — tests/test_search_provider_collision.py: gemini + has_mcp_funcs → onprem_fallback; gemini + vLLM → onprem_fallback (unchanged); gemini + cloud + no MCP → gemini (built-in kept where safe); explicit non-gemini → unchanged. AC6 regression: build the product_research pipeline with cloud_allowed=False → no find_deals, no SearchAPI/Firecrawl, provider unchanged, no built-in+function-tool mix. Gate: pytest tests/test_search_provider_collision.py + full suite — GREEN.

Files: src/sentinel/config/defaults.py — rewrite _P_PRODUCT_RESEARCH_PUBLIC (line 697): step 1 “call find_deals / google_shopping_search for live prices”, step 2 “google_product on each token for multi-seller offers”, steps 3+ generic site: / review searches as fallback; keep _INJECTION_STANCE suffix. Step 7 — Live e2e proof AC5

Rationale: The acceptance proof — a real buying query yields priced products end-to-end on the isolated server.
